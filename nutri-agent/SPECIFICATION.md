## (клиент-центричная модель: клиент → обращения во времени)

> Изменения v3 относительно v2.1 зафиксированы по итогам тестирования прототипа.
> Полный журнал решений и план доработок — в

> Бенчмарк по аналогам (Practice Better, Nutrium, NutriAdmin, Healthie) — там же.

## 1. Две роли — два интерфейса

### Нутрициолог
Цель: вести **клиентов** (а не разрозненные кейсы), загружать свои методички,
анализировать данные в диалоге с ИИ, готовить и отправлять рекомендации.
Видит историю клиента по обращениям и динамику.

### Клиент
Цель: заполнить запрос, опросник и дневник питания прямо в кабинете, загрузить
анализы, по готовности отправить данные нутрициологу, получить рекомендации.

---

## 1.1. Ключевая модель: клиент → обращения

Корневая сущность — **клиент** (обезличенный псевдоним: «Вера Т.», «Юля Б.»).
У клиента во времени несколько **обращений** (бывших «кейсов») — рабочих единиц
с собственным запросом, опросником, дневником, файлами и рекомендациями.

- Стабильные данные (пол, год рождения → возраст, аллергии, хрон. диагнозы)
  живут на уровне клиента и переносятся в новое обращение (carry-forward).
- Запрос / опросник / дневник нового обращения — всегда **свежие**.
- Прошлые обращения доступны нутрициологу для сравнения (динамика веса/ИМТ,
  зон опросника, анализов), но не подмешиваются в свежие данные.

```
Клиент (псевдоним, пол, год рождения)
  └── Обращение 1 (дата, запрос, рост/вес→ИМТ, опросник, дневник, файлы, рекомендации)
  └── Обращение 2 (новый запрос, новые данные ...)
```

Соответствие MVP-схеме: `client_profiles` ↔ клиент, `cases` ↔ обращение
(см. [NUTRI_AGENT_MVP_ARCHITECTURE.md](../NUTRI_AGENT_MVP_ARCHITECTURE.md)).

## 2. Навигация

```
ГЛАВНАЯ СТРАНИЦА
├── «Я нутрициолог» → ПАНЕЛЬ НУТРИЦИОЛОГА
│   ├── Сайдбар: список КЛИЕНТОВ (псевдонимы)
│   ├── Выбор клиента → ПРОФИЛЬ + СПИСОК ОБРАЩЕНИЙ + ДИНАМИКА
│   │   └── Выбор обращения → ДИАЛОГ ДИАГНОСТИКИ + материалы обращения
│   │       └── «+ Новое обращение» (чистое, профиль переносится)
│   ├── Методички (загрузка своих PDF)
│   ├── Кнопка «Отправить рекомендации клиенту»
│   └── ← На главную
│
└── «Я клиент» → ЛИЧНЫЙ КАБИНЕТ КЛИЕНТА (его текущее обращение)
    ├── Запрос (с чем обращается)
    ├── Опросник (заполнить прямо здесь; результаты видит только нутрициолог)
    ├── Дневник питания (заполнить прямо здесь)
    ├── Загрузить анализы (PDF/JPG)
    ├── Кнопка «Данные готовы — отправить нутрициологу»
    └── Мои рекомендации (после отправки нутрициологом)
```

## 3. Методички нутрициолога

База знаний **пуста по умолчанию** — нутрициолог загружает СВОИ методички (PDF).
Продукт рассчитан на специалистов разных школ, поэтому мы не поставляем чужие
методички как стандартную базу. Агент использует только загруженные источники;
если данных нет — помечает 🟡 «нет в загруженных источниках».

Формат: как в чате GPT — нутрициолог грузит методички своей школы.

Как работает:
- Вкладка «Методички» в панели нутрициолога
- Drag-and-drop PDF-файлов
- Агент ссылается на методичку и страницу: «📄 стр.18»
- Можно переключаться между методичками во время анализа
- Если методичка не читается — сообщение об ошибке и предложение перезагрузить

## 4. Запрос, опросник и дневник — прямо в кабинете

### Запрос (на каждое обращение)
Свободное поле: с чем человек обращается, как сам это формулирует, основные
жалобы. Свежее для каждого обращения, видно обеим ролям, входит в стартовую
карту параметров агента.

### Шапка опросника (профиль обращения)
Перед секциями опросника — антропометрия обращения:

| Поле | Источник | Примечание |
|------|----------|------------|
| Пол | профиль клиента (автоподстановка) | стабильный |
| Возраст | из года рождения клиента | считается |
| Рост | вводится в обращении | для ИМТ и динамики |
| Вес | вводится в обращении | для ИМТ и динамики |
| **ИМТ** | вес / (рост_м)² | **только цифра, без категории** |

ИМТ показываем числом. Категорию/расшифровку («норма», «ожирение») НЕ выводим —
её озвучивает нутрициолог, чтобы не пугать клиента «диагнозом».

### Опросник клиента
Полный интегральный опросник PreventAge (орг. часть + клиническая часть,
системы 1–11), реализован в прототипе. Клиент заполняет прямо в интерфейсе,
в несколько подходов.

**Приватность результатов:** клиент видит только вопросы. Баллы, зоны
(низкая/средняя/высокая) и флаг «требует внимания» видит только нутрициолог
(аналог скрытой страницы «РЕЗУЛЬТАТ КЛЮЧ» в исходном xlsx).

Орг. часть (примеры секций): Питание (вредные продукты + полезные привычки),
Вредные субстанции, Образ жизни, Препараты. Клиническая часть: системы 1–11
(ЖКТ, печень, нутриенты, углеводный обмен, надпочечники, щитовидная,
сердечно-сосудистая, почки). Подсчёт по секциям и пороги зон — из ключа.

Интерфейс: кнопки выбора без числовых префиксов, автоматический посекционный
подсчёт (виден нутрициологу).

### Дневник питания
Клиент заполняет прямо в интерфейсе.

Поля на каждый приём пищи:
- Время приёма
- Что ел (текст)
- Способ приготовления
- Объём порции
- Самочувствие после

Дополнительно:
- Вода (сколько и когда пили)
- Лекарства / витамины

Интерфейс: таблица с днями недели, кнопка «+ приём пищи».

## 5. Режим диалога (как в GPT)

Это главная фича, которой не было в первой версии.

### Как работает диалог

Агент ведёт нутрициолога по этапам, а не просто показывает вкладки.

Пример:
```
Агент: «Все данные получены. Перехожу к Этапу 1 — таблица параметров.
        Подтверди: [1] Показать сразу  [2] Загрузить ещё файлы»

Нутрициолог: [нажимает 1]

Агент: [выдаёт таблицу с оптимумами и статусами]
       «Этап 1 завершён. 54 показателя обработано.
        Отклонений: 12. Перехожу к Этапу 2 — выявление отклонений.
        Подтверди: [1] Продолжить  [2] Сменить методичку»

Нутрициолог: [нажимает 2 — загружает другую методичку]

Агент: «Методичка Орлинской загружена. Пересобираю таблицу...»
```

### Элементы диалога
- Окно чата — основная область (вместо вкладок)
- Сообщения агента — с маркерами ✅⚠🟡 и ссылками на методички
- Кнопки подтверждения [1][2] (как в оригинальном агенте)
- Нутрициолог может: писать ответы, загружать файлы, менять методички
- Агент объясняет ошибки («почему не читалась методичка»)

### Чего нет в диалоге (в отличие от GPT)
- Агент не выдумывает значения — только из загруженных методичек
- Все гипотезы с пометкой «предположение»
- При недостатке данных — пометка «нет данных», а не додумывание

## 6. Поток данных

Передача — **явными кнопками**, а не пассивным флагом. Клиент заполняет в
несколько подходов и сам решает, когда данные готовы.

```
КЛИЕНТ                                  НУТРИЦИОЛОГ
────────────                            ──────────────────
Заполняет запрос/опросник/дневник       (статус черновой, не дёргаем)
Загружает анализы по мере получения
[Кнопка: «Данные готовы —
 отправить нутрициологу»] ─────────────→ Отметка «<Псевдоним> отправил(а)
                                          данные на разбор» (уведомление)
                                         Загружает методички (PDF)
                                         Запускает ДИАЛОГ с агентом (Этапы 1→…)
                                         Готовит черновик рекомендаций
                  ┌──────────────────── [Кнопка: «Отправить
                  │                       рекомендации клиенту»]
Видит «Мои        ▼
рекомендации» ◄── (видимы только после кнопки)
```

Прогресс-бары (опросник/дневник/файлы) остаются справочными; сигнал «можно
приступать к разбору» даёт именно кнопка клиента. Клиент может дополнить данные
и переотправить.

## 8. Динамика по обращениям

В карточке клиента нутрициолог видит историю обращений и базовую динамику:
тренд веса/ИМТ, изменение зон опросника от обращения к обращению, доступ к
анализам прошлых обращений как контекст. (Перенято у Nutrium/Practice Better:
антропометрия как временной ряд, а не один снимок.)

## 9. Границы прототипа (Часть A) и план Части B

**Сейчас (Часть A, прототип):** single-page HTML/CSS/JS, состояние в
`localStorage`, без бэкенда. Привязка клиента имитируется псевдонимом/кодом;
«уведомления» — это отметки в интерфейсе по кнопкам-передачам.

**Часть B (после получения ключа OpenAI):**
- Серверный AI-черновик `generateDraft` на **OpenAI GPT-4o** (локальный
  Node-сервер / Supabase Edge Function). AI не вызывается из браузера.
- Реальная привязка клиента: аккаунты + magic-link/код, изоляция доступа (RLS),
  Supabase Storage для файлов, серверная экстракция текста методичек для
  grounding (GPT-4o не читает PDF нативно).
- Реальные уведомления (e-mail/push).

Принципы безопасности сохраняются: агент не выдумывает значения (только из
загруженных методичек), помечает гипотезы и нехватку данных, финальные
формулировки проверяет специалист; клиент видит только проверенную версию.
