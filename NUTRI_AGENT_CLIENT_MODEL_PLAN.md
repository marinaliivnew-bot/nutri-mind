## 1. Контекст: что изменилось после тестирования

Часть A (реальный опросник PreventAge, своя база знаний, приватность результатов,
удаление демо-кейсов) собрана. Тестирование показало, что модель «кейс как
корень» не отражает реальную работу: у нутрициолога есть **клиенты**, и каждого
он ведёт во времени через несколько **обращений**.

### Журнал решений (зафиксировано с пользователем 29.06.2026)

1. **Корень — клиент, а не кейс.** Сайдбар показывает клиентов по псевдониму
   («Вера Т.», «Юля Б.»), без реальных ПД.
2. **У клиента несколько обращений во времени** (Вариант B): разные запросы
   (похудеть / набрать), меняющиеся привычки, диагнозы. Обращение — рабочая
   единица (бывший «кейс»).
3. **Поле «Запрос»** на каждое обращение: с чем пришёл, как сам формулирует,
   + жалобы. Свежее для каждого обращения.
4. **Динамика без потери свежести:** прошлые обращения доступны нутрициологу для
   сравнения (вес/ИМТ, анализы, зоны опросника), но запрос / опросник / дневник
   нового обращения заполняются заново.
5. **Шапка опросника:** пол, возраст, рост, вес и **ИМТ — только цифра**
   (без категории/«диагноза»; расшифровку даёт нутрициолог сам).
6. **Передача — явными кнопками, а не пассивным флагом:**
   - клиент: «Данные готовы — отправить нутрициологу»;
   - нутрициолог: «Отправить рекомендации клиенту».
   Прогресс-бары остаются справочными; сигнал «можно приступать» = кнопка.
7. **База знаний пуста по умолчанию** — нутрициолог грузит свои методички (Часть A, сделано).
8. **Приватность опросника** — клиент видит только вопросы; баллы и зоны видит нутрициолог (Часть A, сделано).
9. **Демо-кейсы убраны** — старт с пустого списка (сделано).

## 2. Бенчмарк: как это сделано у других

Источники: Practice Better, Nutrium, NutriAdmin, Healthie (ссылки — раздел 7).

### Подтверждает наши решения
- **Явная отправка форм клиентом** — в NutriAdmin клиент «submit option to send
  responses directly to your provider». = наша кнопка «Отправить нутрициологу».
- **Двусторонние уведомления** — клиента уведомляют, когда ему что-то отправили;
  нутрициолога — когда форма заполнена. = две кнопки-передачи.
- **Заполнение в несколько подходов** — «save progress across multiple sessions».
- **Клиент как центральная запись во времени** — questionnaires, health history,
  anthropometrics, documents, reports «associated with clients over time».
  = модель «клиент → обращения».

### Фишки, которые перенимаем (умнее, чем просто ИМТ)
- **Антропометрия как временной ряд.** Лидеры ведут серию замеров (вес, талия и
  пр.) с трендом, а не один ИМТ. → шаг 4: вес/ИМТ по обращениям как мини-тренд.
  ИМТ оставляем цифрой, но показываем динамику.
- **«Carry-forward профиль + свежая оценка».** Стабильные данные клиента (пол,
  аллергии, хрон. диагнозы) переносятся в новое обращение автоматически, а
  запрос/опросник/дневник остаются чистыми. → при «Новом обращении».
- **Чек-лист «что ещё не получено»** в карточке обращения — но финальный сигнал
  даёт кнопка клиента, а не автостатус.
- **Magic-link / e-mail приглашение** как механизм привязки в реальном продукте
  (Часть B). В прототипе — код/псевдоним ради обезличивания.
- **Библиотека опросников + кастомизация** — у нас опросник фиксированный
  (PreventAge); кастомные опросники — бэклог Части B.

## 3. Целевая модель данных (прототип)

```
Client (верхний уровень, стабильная личность)
├── id, alias (псевдоним), gender, birthYear (→ возраст)
├── carryForward: стабильные заметки (аллергии, хрон. диагнозы) — опционально
└── обращения: [appealId, appealId, ...]   // несколько во времени

Appeal / Обращение (бывший «кейс» — рабочая единица)
├── id, clientId, createdAt, title («Обращение 03.2026»), status
├── request (Запрос: формулировка клиента + жалобы)   // свежее
├── anthro: { height, weight }  → ИМТ считается на лету (цифра)
├── questionnaire (свежий)
├── diaryEntries (свежий)
├── uploadedFiles { client, nutritionist }
├── nutritionistSources (методички)
├── agentDialog, draftReport
└── handoff: { clientSubmitted, clientSubmittedAt,
               recommendationsSent, recommendationsSentAt }
```

Технически слой «обращение» = текущий per-case бандл (минимум ломки): сверху
добавляем слой «клиент» (профиль + список обращений) и поля `request`, `anthro`,
`handoff`. Статусы обращения двигают кнопки-передачи, а не автофлаг.

Совместимость с MVP-схемой: `client_profiles` ↔ Client, `cases` ↔ Appeal
(в `cases` добавятся `request`, `height`, `weight`, поля handoff). Перенос в
Supabase из [NUTRI_AGENT_MVP_ARCHITECTURE.md](NUTRI_AGENT_MVP_ARCHITECTURE.md)
остаётся валидным.

## 4. Шаги доработки (прогресс)

### Шаг 1. Слой «клиент» над обращениями
**Статус:** ✅ выполнено

- [x] Сущность `Client` (alias, gender, birthYear) над обращениями.
- [x] Сайдбар показывает клиентов (псевдоним + метка: возраст · цель).
- [x] Выбор клиента → профиль + список его обращений с датами и статусами.
- [x] «+ Новое обращение» создаёт чистое обращение, перенося профиль клиента
      (carry-forward), но не копируя запрос/опросник/дневник.
- [x] «+ Добавить клиента» (бывш. «Добавить кейс»).
- [x] Миграция состояния (bump ключа): старые плоские обращения → под авто-клиента
      или чистый старт.

**Приёмка:** в сайдбаре клиенты; у клиента несколько обращений, данные не
смешиваются; новое обращение чистое с автоподстановкой профиля; всё переживает
перезагрузку.

### Шаг 2. Шапка опросника + поле «Запрос»
**Статус:** ✅ выполнено

- [x] Шапка опросника: пол, возраст (из профиля), рост, вес.
- [x] **ИМТ = вес / (рост_м)²**, только число (1 знак), без категории.
- [x] Поле «Запрос» на обращение (формулировка клиента + жалобы); видно обеим
      ролям, входит в карту параметров агента.
- [x] Рост/вес сохраняются в `anthro` обращения (для динамики, шаг 4).

**Приёмка:** ИМТ считается на лету, категория не показывается; запрос сохраняется,
виден обеим ролям, попадает в стартовое сообщение агента.

### Шаг 3. Кнопки-передачи вместо флага
**Статус:** ✅ выполнено

- [x] Кнопка клиента «Данные готовы — отправить нутрициологу» →
      `handoff.clientSubmitted`, статус «данные отправлены».
- [x] У нутрициолога отметка «<Псевдоним> отправил(а) данные на разбор»
      (это уведомление). Клиент может дополнить и переотправить.
- [x] Кнопка нутрициолога «Отправить рекомендации клиенту» → видимость в кабинете
      клиента (поверх статусов `approved_for_client`/`sent_to_client` Этапа 4).
- [x] Убрать «отмашку» по автостатусу из логики уведомления.

**Приёмка:** пока клиент не нажал «Отправить» — нутрициолога не дёргает; после
нажатия появляется явная отметка; рекомендации видны клиенту только после кнопки.

### Шаг 4. Базовая динамика по обращениям
**Статус:** ✅ выполнено

- [x] В карточке клиента — список обращений со снимками (дата, вес, ИМТ,
      заполненность опросника).
- [x] Мини-тренд веса/ИМТ по обращениям.
- [x] Сравнение зон опросника между обращениями.
- [x] Доступ нутрициолога к данным прошлых обращений как контекст.

**Приёмка:** видно изменение веса/ИМТ и зон от обращения к обращению; новые
обращения остаются свежими (старые данные — только для сравнения).

## 5. Definition of Done (v3)

- Сайдбар — клиенты; у клиента несколько обращений во времени.
- Новое обращение чистое, профиль клиента переносится автоматически.
- В опроснике есть пол/возраст/рост/вес и ИМТ-цифра без категории.
- Есть поле «Запрос» на каждое обращение.
- Передача данных и рекомендаций — явными кнопками, без пассивного флага.
- Видна базовая динамика по обращениям.
- Всё переживает перезагрузку; данные обезличены; это по-прежнему прототип без
  реального backend.
