# План превращения NutriMind в рабочий прототип

Дата: 26 июня 2026

## 1. Цель

Сделать текущий single-page прототип NutriMind не только кликабельным, но и логически рабочим: действия пользователя должны менять выбранный кейс, клиентские материалы должны быть связаны с кабинетом нутрициолога, а диалог с Nutri Intelligence Agent должен опираться на состояние конкретного кейса.

На первом этапе это остается локальным прототипом без бэкенда. Основная задача — собрать правильную продуктовую механику и модель данных, чтобы затем без потерь перенести ее в MVP с сервером, пользователями, файлами и реальным AI.

## 2. Принципы

- Сначала рабочая логика, потом инфраструктура.
- Все данные в демо должны быть обезличены.
- Клиент не видит профессиональные УТП и черновики до проверки специалистом.
- Nutri Intelligence Agent готовит черновики, но не принимает финальные решения.
- Любое действие в интерфейсе должно иметь понятный результат: сохранение, изменение статуса, обновление прогресса, появление материала в другом кабинете.
- Формулировки остаются безопасными: не диагнозы, не лечение, не назначения.

## 4. Этап 2. Сделать клиентские материалы реально связанными с кейсом

**Статус:** выполнено.

**Проверка:** в кабинете нутрициолога (вкладка «Профиль») добавлена живая панель материалов выбранного кейса — статус опросника (не начат / N из total / заполнен), дней дневника, количества и списка файлов клиента и статуса рекомендаций; фейковая статистика заменена на реальные данные. В шапке клиентского кабинета добавлен индикатор текущего кейса. Представления перерисовываются при переключении ролей в `enterApp()`. Синтаксис встроенного JS проверен через `new Function(...)`; поток данных (клиент заполняет опросник/дневник/файлы → статус обновляется у нутрициолога, изоляция между кейсами, восстановление после перезагрузки) проверен Node smoke-тестом на `vm` + DOM-заглушке (16 проверок пройдено).

**Цель:** клиентский кабинет должен заполнять материалы конкретного выбранного или демо-кейса, а нутрициолог должен видеть эти изменения.

### Что сделать

- Связать ответы опросника с `caseId`.
- Связать дневник питания с `caseId`.
- Связать воду, добавки/лекарства и заметки с `caseId`.
- Связать загруженные клиентом файлы как метаданные с `caseId`.
- В кабинете нутрициолога показывать статус материалов выбранного кейса:
  - опросник не начат / частично / заполнен;
  - дневник: количество заполненных дней;
  - файлы: количество и список;
  - рекомендации: черновик / на проверке / готово клиенту.

### Критерии приемки

- Клиент заполняет опросник, и прогресс меняется у нутрициолога.
- Клиент добавляет запись дневника, и она учитывается в выбранном кейсе.
- Клиент загружает файл, и он появляется в материалах кейса у нутрициолога.
- При переключении кейсов данные не смешиваются.

## 5. Этап 3. Сделать рабочий сценарий диалога агента

**Статус:** выполнено.

**Проверка:** история диалога (`agentDialog[caseId].history`) стала источником истины — каждое сообщение агента и нутрициолога пишется в неё и перерисовывается при переключении кейсов и после перезагрузки (раньше восстанавливался только номер этапа). Стартовое сообщение `buildIntroMessage()` генерируется из заполненности кейса (опросник/дневник/файлы/источники + список пробелов). Действия в диалоге меняют `draftReport.stageStatus` по этапам, решения по гипотезам (Принять/Уточнить/Отклонить) сохраняются в `draftReport.hypotheses`, а статус черновика переходит в `needs_review` на этапе рекомендаций (отражается в панели материалов нутрициолога). «Показать источник» добавляет сообщение без смены этапа, «Начать заново» очищает историю, повторный клик по использованной кнопке игнорируется. Синтаксис проверен через `new Function(...)`; поведение (история, динамический старт, смена статусов, решения по гипотезам, изоляция между кейсами, восстановление после перезагрузки, источник/рестарт/защита от повторного клика) проверено двумя Node smoke-тестами на `vm` + DOM-заглушке (30 проверок пройдено).

**Цель:** диалог должен быть связан с материалами кейса и менять состояние черновика.

### Что сделать

- Хранить историю диалога по `caseId`.
- Добавить генерацию стартового сообщения на основе текущей заполненности кейса.
- Действия в диалоге должны менять статус этапа:
  - входные данные проверены;
  - таблица параметров открыта;
  - пробелы данных отмечены;
  - гипотезы приняты / уточнены / отклонены;
  - черновик рекомендаций подготовлен;
  - клиентская версия готова к проверке.
- Хранить решения нутрициолога по гипотезам.
- Добавить состояние `draftReport` для выбранного кейса.

### Критерии приемки

- Диалог выбранного кейса восстанавливается после перезагрузки.
- Нажатия «Принять», «Уточнить», «Отклонить», «Показать источник» меняют историю и статус.
- Черновик рекомендаций зависит от текущего кейса.
- У другого кейса своя история диалога и свой черновик.

## 6. Этап 4. Сделать черновик отчета и клиентскую версию

**Статус:** выполнено.

**Проверка:** вкладка «Рекомендации» нутрициолога перестала быть статичной — теперь это динамический черновик из `draftReport` выбранного кейса: сводка по кейсу, пробелы данных (вычисляются из заполненности), решения по гипотезам из диалога, редактируемый черновик рекомендаций и отдельные заметки нутрициолога (не видны клиенту), бейдж статуса и кнопки перехода по статусам (`draft` → `needs_review` → `approved_for_client` → `sent_to_client`, с возвратом на доработку). Клиентская вкладка «Рекомендации» показывает контент только при статусе `approved_for_client`/`sent_to_client`; иначе — заглушка «Готовятся» без гипотез и заметок. «Одобрить для клиента» формирует `clientVersion` из черновика рекомендаций (без внутренних заметок). Стат-карточка «Рекомендации» в кабинете клиента и панель материалов нутрициолога отражают актуальный статус. Экспорт отчёта дополнен сводкой, пробелами, решениями по гипотезам, черновиком рекомендаций, заметками и клиентской версией выбранного кейса. Синтаксис встроенного JS проверен через `new Function(...)`; поведение (рендер черновика, скрытие от клиента до одобрения, генерация клиентской версии без заметок, смена/возврат статусов, изоляция между кейсами, восстановление после перезагрузки) проверено Node smoke-тестом на `vm` + DOM-заглушке (28 проверок пройдено).

**Цель:** отделить рабочий черновик нутрициолога от версии, которую может увидеть клиент.

### Что сделать

- Создать структуру `draftReport`:
  - summary;
  - dataGaps;
  - hypotheses;
  - nutritionistNotes;
  - recommendationDraft;
  - clientVersion;
  - status.
- Добавить статусы:
  - `draft`;
  - `needs_review`;
  - `approved_for_client`;
  - `sent_to_client`.
- В клиентском кабинете показывать рекомендации только после статуса `approved_for_client` или `sent_to_client`.
- В экспорте отображать данные выбранного кейса и актуальный статус.

### Критерии приемки

- Клиент не видит рабочие гипотезы и профессиональные черновики.
- Нутрициолог видит черновик, может редактировать заметки и менять статус.
- После перевода в клиентскую версию рекомендации появляются в кабинете клиента.
- Экспорт отчета соответствует выбранному кейсу.

## 7. Этап 5. Добавить импорт и экспорт демо-состояния

Применение состояния (`applyDemoState`) идёт через `normalizeDemoState` + общий `refreshAllViews()` (вынесен из `resetDemoData`) + сохранение. ## 11. Минимальный Definition of Done для рабочего локального прототипа

- Один выбранный кейс управляет всеми данными на экране.
- Клиентские действия отражаются в кабинете нутрициолога.
- Диалог агента сохраняет историю и решения по гипотезам.
- Черновик отчета отделен от клиентской версии.
- Все состояние переживает перезагрузку страницы.
- Демо можно экспортировать и импортировать.
- Интерфейс честно показывает, что это прототип без реального backend и без реальных персональных данных.
